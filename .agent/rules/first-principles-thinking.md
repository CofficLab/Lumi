# 第一性原理思考规范

> 本规范定义了 Lumi 项目中使用第一性原理（First Principles）进行问题分析和技术决策的指导方针。

---

## 核心原则

**从基本真理出发，通过逻辑推导解决问题，而不是依赖类比、经验或既有结论。**

---

## 1. 思维模式

### 1.1 第一性原理 vs 类比思维

| 场景 | 类比思维（避免） | 第一性原理（推荐） |
|-----|----------------|------------------|
| 技术选型 | "别人都在用这个框架" | "我的问题本质是什么？这个技术如何从根本上解决它？" |
| 性能优化 | "加缓存、升级服务器" | "时间花在哪里？瓶颈是 CPU/IO/网络？" |
| 代码重构 | "按最佳实践重写" | "这段代码的核心职责是什么？当前实现是否偏离？" |
| 解决报错 | "复制粘贴解决方案" | "从数据流、执行流程的基本原理出发定位根源" |

### 1.2 追问本质的方法

遇到问题时，至少问 3 层"为什么"：

```
表面问题：按钮点击没反应

第 1 层：为什么没反应？→ 事件处理函数没执行
第 2 层：为什么没执行？→ 事件监听器没有正确绑定
第 3 层：为什么没绑定？→ 视图生命周期中绑定代码在条件分支内，条件不满足
根源：绑定逻辑应该无条件执行，条件检查应在处理函数内部
```

## 2. 编码实践

### 2.1 理解底层原理

**不要死记硬背 API，要理解其工作机制。**

```swift
// ❌ 避免：只记住用法
await fetch(url).then(res => res.json())

// ✅ 推荐：理解事件循环和 Promise 机制
// → 能正确处理并发、错误边界、竞态条件
```

### 2.2 技术决策

**从问题本质出发选择技术方案。**

```swift
// ❌ 避免：盲目追新
"SwiftUI 是最新的，所以用 SwiftUI"

// ✅ 推荐：基于需求分析
"""
问题本质：
1. 需要复杂的自定义动画 → SwiftUI 动画系统是否支持？
2. 需要精细的触摸控制 → SwiftUI 的 gesture 是否足够？
3. 团队熟悉度如何？学习成本 vs 收益？

结论：根据具体问题选择最合适的方案
"""
```

### 2.3 调试方法

**从基本原理推导问题根源。**

```swift
// 问题：数据没有更新到 UI

// ❌ 避免：盲目试错
// 试试加 @State，试试加 .onChange，试试...

// ✅ 推荐：系统分析
"""
1. SwiftUI 更新机制：@Published → objectWillChange → View 重新计算
2. 检查数据源：@Published 属性是否真正改变？
3. 检查线程：更新是否在主线程？
4. 检查引用：是否引用了正确的对象实例？
5. 检查条件：body 中是否有条件阻止了更新？
"""
```

## 3. 架构设计

### 3.1 从本质推导架构

**设计系统时先问基本问题。**

```
设计一个缓存系统：

1. 缓存的本质是什么？→ 用空间换时间
2. 核心矛盾是什么？→ 一致性与性能的权衡
3. 基本操作有哪些？→ 读、写、失效、更新
4. 约束条件是什么？→ 内存限制、过期策略、并发安全
5. 由此推导：
   - 淘汰策略：LRU / LFU / FIFO？
   - 一致性：强一致 / 最终一致？
   - 并发：锁 / Actor / 无锁？
```

### 3.2 简化实现

**找到最简单的正确实现。**

```swift
// ❌ 避免：过度设计
class DataCacheManager {
    private let lock = NSLock()
    private let queue = DispatchQueue(label: "...")
    private var cache: [String: CachedItem] = [:]
    private var accessOrder: [String] = []
    // ... 100+ 行复杂逻辑
}

// ✅ 推荐：从本质出发
"""
核心需求：
1. 存储键值对 → Dictionary
2. 线程安全 → Actor（Swift 原生支持）
3. 过期清理 → 定期检查 timestamp

最简单的正确实现优先，必要时再优化
"""
```

## 4. 问题排查流程

### 4.1 标准流程

```
1. 定义问题
   - 现象是什么？
   - 预期行为是什么？
   - 何时开始出现的？

2. 拆解假设
   - 列出所有关于这个问题的既有认知
   - 哪些是事实？哪些是假设？

3. 追问本质
   - 至少问 3 层"为什么"
   - 直到无法再分解的基本原理

4. 验证推导
   - 基于原理设计验证实验
   - 用日志/断言/单元测试确认

5. 修复方案
   - 从根源修复，而不是打补丁
   - 确保修复不引入新问题
```

### 4.2 示例

```
问题：应用启动时内存占用过高

1. 定义问题
   - 现象：启动后内存 200MB+
   - 预期：启动后内存 50MB 左右
   - 时间：v2.0 版本开始

2. 拆解假设
   - 假设：新功能加载了大量数据 ✓ 验证
   - 假设：图片缓存策略改变 ✓ 验证
   - 假设：有内存泄漏 ✗ 待验证

3. 追问本质
   - 为什么内存高？→ 加载了大量图片
   - 为什么加载大量图片？→ 预加载策略改为全量预加载
   - 为什么全量预加载？→ 为了避免滚动时卡顿
   - 本质矛盾：启动性能 vs 滚动性能

4. 验证推导
   -  Instruments 检查内存分配
   - 确认图片加载是主要消耗

5. 修复方案
   - 改为分页预加载 + 滚动时懒加载
   - 从根源平衡两个性能指标
```

## 5. 代码审查检查清单

审查代码时，用第一性原理思考：

### 5.1 必要性检查

- [ ] 这段代码解决的核心问题是什么？
- [ ] 是否有更简单的实现方式？
- [ ] 是否引入了不必要的复杂性？

### 5.2 原理检查

- [ ] 实现是否符合底层原理（如 SwiftUI 更新机制、Swift 并发模型）？
- [ ] 是否理解了使用的 API 的工作方式？
- [ ] 是否有未经验证的假设？

### 5.3 可维护性检查

- [ ] 代码是否表达了"为什么"而不仅仅是"做什么"？
- [ ] 注释是否解释了设计决策的原因？
- [ ] 是否便于后续基于原理进行修改？

## 6. 学习与应用

### 6.1 培养习惯

1. **多问"为什么"**：至少问 3 层，直到基本原理
2. **读源码**：理解框架/库的实现原理
3. **手写基础实现**：自己实现一遍关键数据结构或算法
4. **画图分析**：把抽象问题可视化（数据流、状态转换）
5. **复盘总结**：解决问题后反思本质原因

### 6.2 适用场景

| 优先级 | 场景 | 建议 |
|-------|------|------|
| 🔴 高 | 核心算法、关键架构决策 | 必须用第一性原理 |
| 🟡 中 | 性能优化、复杂 bug 排查 | 推荐用第一性原理 |
| 🟢 低 | 简单任务、重复工作 | 可以用经验/类比提效 |

### 6.3 注意事项

第一性原理虽然强大，但需要平衡效率：

```
✅ 正确用法：
- 核心技术问题 → 深入分析本质
- 关键架构决策 → 从原理推导
- 复杂 bug 排查 → 系统性分析

❌ 过度使用：
- 简单 CRUD 操作 → 无需过度分析
- 成熟的最佳实践 → 可以直接采用
- 紧急修复 → 先解决再优化
```
