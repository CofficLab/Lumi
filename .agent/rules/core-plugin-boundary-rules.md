# 内核与插件边界规范

> 本规范定义 Lumi 中**内核（Core）**与**插件（Plugins）**之间的依赖方向：保证内核可独立演进、插件可插拔，避免循环依赖与隐式耦合。

---

## 术语

| 术语 | 含义 |
|------|------|
| **内核** | `LumiApp/Core/` 下的通用能力，以及不属于任何插件目录的应用引导、共享协议与基础设施（如 `RootViewContainer`、`SuperPlugin`、`PluginVM`、核心 `Service` 等）。以「不挂在某个具体插件名下」为界。 |
| **插件** | 位于 `LumiApp/Plugins/`、`LumiApp/Plugins-Agent/`、`LumiApp/Plugins-Tool/`、`LumiApp/Plugins-LLM/` 等插件目录中的代码；每个插件自洽，通过 `SuperPlugin` 等机制注册。 |

若对某文件归属有疑问，以**物理路径是否在上述插件目录下**为准。

---

## 规则一：内核不得依赖插件

**内核代码禁止引用、调用或导入任何插件实现中的类型与符号。**

包括但不限于：

- 不得 `import` 插件模块或引用插件内的 `struct` / `class` / `enum` / 扩展。
- 不得在 Core 中硬编码某个插件的类名、工具类型名或插件专属资源路径。
- 不得在内核中假设「某插件一定存在」；能力通过协议、注册表、可选钩子或依赖注入由上层组装。

若内核需要扩展点，应在内核中声明**抽象**（协议、注册接口、事件名），由应用在启动或 `PluginVM` 加载插件后，把实现挂接上去。

---

## 规则二：插件不得依赖其他插件

**任一插件代码禁止引用、调用或导入其他插件中的类型与符号。**

包括但不限于：

- 不得跨插件目录引用（例如 `Plugins-Agent/A` 引用 `Plugins/B` 中的类型）。
- 不得假设其他插件已启用；插件之间**禁止**直接耦合。

若需要在插件间协调：

- 通过**内核提供的**事件、通知、`PluginVM` / 容器上已有的服务与协议转发；
- 或通过用户显式要求的**共享中立层**（仅当项目已存在且规范允许时；新能力优先加在内核抽象上）。

---

## 设计推论

- 共享逻辑若被多个插件需要，应**下沉到内核**（或已存在的共享模块），而不是让插件 A 依赖插件 B。
- 新功能默认落在**单一插件**内；只有确认会被多处复用且不破坏边界时，再抽到内核。

---

## 检查清单（变更代码时自问）

- [ ] Core 中是否出现了 `Plugins*` 路径下的类型引用？
- [ ] 插件 A 是否 import 或使用了插件 B 的源码？
- [ ] 若必须「像插件一样扩展」，是否已通过内核协议/注册表而非直接类型依赖？
