## 目标

在当前主线 (`dev` 分支) 上,引入一个**全新的 `LumiPlugin` 钩子函数**,让插件在 LLM 请求组装前**直接修改/注入**消息(主要是追加 system prompt),**完全脱离 4.19.0 的 `LumiSendMiddleware` struct 形态**。同时清理旧的 `SendMiddleware` 协议,只保留新钩子。

## 关键设计决策(已与用户确认)

| 决策点 | 选择 | 理由 |
|---|---|---|
| 钩子形态 | 新增 `LumiPlugin` 实例方法 `willSendToLLM(kernel:messages:)` | 插件直接改 messages,无需中间件 struct |
| 缓存拼接位置 | **单条合并 system 消息,放在 `messages[0]`** | 对 OpenAI 自动 cache 前缀匹配最友好;Anthropic adapter 已会自动 join,等价 |
| 旧协议处理 | **废弃 `SendMiddleware`/`SendMiddlewareProviding`,只保留新钩子** | 简化协议面,避免双轨制 |
| 协议挂在哪 | **挂在 `LumiPlugin` 上**(而非独立协议) | "新方案 = 新钩子函数",但沿用 LumiPlugin 作为唯一注册源 |
| 测试 | 暂不写 | 用户要求 |

```swift
// 收集所有插件的 willSendToLLM 贡献,按插件 order 顺序串行应用
var preparedMessages = history
for plugin in (kernel.pluginManager?.allPlugins ?? []) {
    guard plugin.policy.shouldRegister else { continue }
    preparedMessages = await plugin.willSendToLLM(kernel: kernel, messages: preparedMessages)
}

// 拼接策略:如果 preparedMessages[0] 已是 system 消息 → 合并 fragments;
// 否则 → 在首位插入一条 system 消息
let fragments = preparedMessages.filter { $0.role == .system }.map(\.content)
if !fragments.isEmpty {
    let merged = fragments.joined(separator: "\n\n")
    let nonSystem = preparedMessages.filter { $0.role != .system }
    preparedMessages = [
        LumiChatMessage(conversationID: conversationID, role: .system, content: merged)
    ] + nonSystem
}

let request = LumiLLMRequest(
    messages: preparedMessages,
    model: model,
    tools: tools,
    imageAttachments: pendingImages
)
```

> 注:`pluginManager` 暴露方式取决于 `LumiKernel` 现有 API;若 `kernel` 不直接暴露插件列表,可能需要新增 `kernel.allPlugins` 访问器,或者通过 `BuiltinPluginManager` 单例。先实施 Step 1-5,在 Step 6 阶段再决定是否需要补充访问器。

## 风险与注意

1. **`kernel.allPlugins` 访问器可能需要新增**:`AgentTurnRunner` 当前只能通过 `kernel.pluginManager?` 间接拿到插件列表(`BuiltinPluginManager`)。需要先确认该访问器已存在;若不存在,在 `LumiKernel+Services.swift` 加一行 `public var pluginManager: (any PluginRegistry)?`
2. **`pluginManager.allPlugins` 在 `MainActor` 上访问**:由于 `AgentTurnRunner` 自身已是 `@MainActor`,无并发问题
3. **每个 agent turn 循环都会重跑钩子**:与 4.19.0 中间件行为一致;插件应自行做幂等保护(若需要)
4. **钩子抛错怎么办**:默认设计为不抛错(`async` 返回新数组)。如插件内部需要抛错,应自行捕获并返回原数组,避免中断整个 turn
