# 恢复 V1 + V2 两个文件树插件到侧栏

## 背景
当前主分支的 `Plugins/EditorFileTreePlugin` (V1, SwiftUI 原生) 和 `Plugins/EditorFileTreeV2Plugin` (V2, NSCollectionView 桥接) 已经被改成了空 stub:

- `policy: .disabled` → 内核不会注册任何东西
- `panelRailTabItems(kernel:) -> [PanelRailTabItem] { [] }` → 即使启用也不返回任何 rail tab

所以 rail 里看不到 "Explorer"(也没有 "Explorer V2")。

4.19.0 旧版这两个插件的文件差异极小 — `TreeView`/`TreeViewV2`、Services、Views 都还在,只是 `EditorFileTreePlugin.swift` 和 `EditorFileTreeV2Plugin.swift` 两个入口被"掏空"了。

## 设计目标
1. 把 V1、V2 两个文件树插件重新接到 `LumiPlugin` 协议
2. 在 rail 中注册两个 tab:`explorer` (V1) 和 `explorer-v2` (V2)
3. 两个都默认启用 (`.optOut`),用户可在设置中独立禁用

## 关键 API 调研结论
- **`LumiPlugin` 协议** (`Packages/LumiKernel/Sources/LumiKernel/Contracts/LumiPlugin.swift`) 是 class-based,`panelRailTabItems(kernel: LumiKernel) -> [PanelRailTabItem]`
- **`PanelRailTabItem`** (`Types/PanelItem.swift:76`) 公开初始化器:`(id:, title:, systemImage:, content:)` — 没有 `order`(`BuiltinPluginManager.swift:140` 会用 plugin 自身的 `order` 覆盖)
- **LumiCore 访问**:`kernel.lumiCore` 返回 `(any LumiCoreProviding)?`,`LumiCoreAccessing` 已经是 `LumiCoreProviding` 的 typealias
- **`TreeView` / `TreeViewV2`** 已经是 current-compatible:`init(lumiCore: LumiCoreAccessing)`,body 读 `lumiCore.projectComponent.currentProject` 和 `@EnvironmentObject EditorContext`
- **rail 渲染链路**:`BuiltinPluginManager` 拿 `panelRailTabItems(kernel:)` → `kernel.panel?.registerPanelRailTabItem(...)` → `AppLayoutView.swift:29` 取 `kernel.panel?.allPanelRailTabItems` → `SimpleRailView` 渲染 (Icons + 标题 + `makeView()`)
- **工作样例**:`ConversationListPlugin.swift:36-46`、`EditorSearchPlugin.swift:38-53` 是已经在用的非 stub 范例

## 实施步骤

### 步骤 1 — 改造 V1 `EditorFileTreePlugin.swift`
文件:`Plugins/EditorFileTreePlugin/Sources/EditorFileTreePlugin.swift`

把当前的 class stub 替换为:
```swift
@MainActor
public final class EditorFileTreePlugin: LumiPlugin {
    public let id = "com.coffic.lumi.plugin.editor-rail-file-tree"
    public let name = "Editor File Tree"
    public let order = 50          // Editor 区域插件给一个合理 order
    public let policy: LumiPluginPolicy = .optOut   // <-- 用户选择:默认启用

    public init() {}

    public func onBoot(kernel: LumiKernel) async throws {}

    public func onReady(kernel: LumiKernel) async throws {
        guard let lumiCore = kernel.lumiCore else { return }
        EditorFileTreePluginBootstrap.bootstrapFromLumiCoreIfNeeded(
            context: lumiCore
        )
    }

    public func panelRailTabItems(kernel: LumiKernel) -> [PanelRailTabItem] {
        guard let lumiCore = kernel.lumiCore else { return [] }
        return [
            PanelRailTabItem(
                id: "explorer",
                title: "Explorer",
                systemImage: "folder"
            ) {
                TreeView(lumiCore: lumiCore)
            }
        ]
    }
    // ...其余 stub 保留(返回空数组)
}
```

要点:
- `EditorFileTreePluginBootstrap.bootstrapFromLumiCoreIfNeeded(context:)` 已存在且签名兼容 (`context: any LumiCoreAccessing`),直接调用
- 把 `Explorer` 文案从 localization 拿 (`LumiPluginLocalization.string("Explorer", bundle: .module)`) — 但 V1 文件已经有 `LumiPluginLocalization` 服务,可以从那里取
- `id`/`name` 保持与 `LumiPluginInfo` 一致(`com.coffic.lumi.plugin.editor-rail-file-tree`)

### 步骤 2 — 改造 V2 `EditorFileTreeV2Plugin.swift`
文件:`Plugins/EditorFileTreeV2Plugin/Sources/EditorFileTreeV2Plugin.swift`

并行结构,只是:
- `id = "com.coffic.lumi.plugin.editor-file-tree-v2"`
- `name = "Editor File Tree V2"`
- `systemImage: "square.grid.2x2.fill"`
- tab id `"explorer-v2"`
- `panelRailTabItems` 返回 `TreeViewV2(lumiCore: lumiCore)`
- 同样调 `EditorFileTreeV2PluginBootstrap.bootstrapFromLumiCoreIfNeeded(context: lumiCore)`
- `policy: .optOut`(用户选择)

### 步骤 3 — Plugin.order 选值
参考 `EditorPanelPlugin` 用 `order = 77`,把 V1 设为 `50` / V2 设为 `51`(或者保留 `0`,由 `BuiltinPluginManager` 自动按注册顺序),确保 rail 上 V1 在 V2 之上。简单起见:V1 `order = 50`,V2 `order = 51`。

### 步骤 4 — Stub 保留
其他贡献方法(`llmProviders`、`chatSection*`、`viewContainers` 等)保持返回空数组,与现有 Plugins 写法一致即可。

### 步骤 5 — 本地化(如需要)
4.19.0 的 V1/V2 都用 `LumiPluginLocalization.string("Explorer", ...)`.当前两个 plugin 都已经有 `Sources/Services/LumiPluginLocalization.swift` 文件可以直接调用。

### 步骤 6 — Build 验证
运行
```
xcodebuild -scheme Lumi -configuration Debug -destination 'platform=macOS' build
```
确认 plugin 编译通过且 rail 注册正常。

## 不需要修改
- `Packages/LumiKernel`(rail 注册 API 已就绪)
- `Packages/FileSystemKit`(`Tree/` 子目录里 `FileTreeStore.swift`、`FileTreeService.swift`、`FileTreeWatcher.swift` 已存在)
- `Plugins/EditorFileTreePlugin/{Models,Views,Services}` 和 V2 同(`TreeView` / `TreeViewV2` 已是 current API)
- `Plugins/EditorFileTreePluginBootstrap.swift` / `EditorFileTreeV2PluginBootstrap.swift`(签名已经匹配 `LumiCoreAccessing` typealias)

## 风险
- **侧栏出现两个 Explorer tab**:用户预期如此(回答中明确选择"V1 和 V2 都 .optOut");V1 标题"Explorer",V2 标题"Explorer V2",不会混淆
- **`Policy.shouldRegister`**: `BuiltinPluginManager` 跳过 `policy == .disabled` 的 plugin,所以改 `.optOut` 后确实会跑到 `panelRailTabItems(kernel:)`
- **`panelRailTabItems` 返回的 view 关闭时是否泄漏状态**:V1/V2 内部的 `lumiCore.projectComponent.currentProject` 已是 Combine 订阅,沿用现成 pattern 不会泄漏

## 验证步骤
1. `xcodebuild ... build` — 整体构建通过
2. 启动 app → editor container → 检查 rail 应同时显示两个 tab:"Explorer"(folder 图标)和"Explorer V2"(grid 图标)
3. 进入设置 → Plugins → 应能单独启用/禁用 V1/V2
4. 选中"Explorer"标签时,V1 应正常展示项目树
5. 选中"Explorer V2"时,V2 `NSCollectionView` 桥接渲染树
